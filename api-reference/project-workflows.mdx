The v2 platform now exposes the core quote-to-project workflows that previously required
public v1 or Angular/internal v1 calls. These endpoints keep the v2 platform model:
bearer authentication, account-scoped `PlatformContext`, public GUID route IDs, release
governance metadata, usage recording, standard error envelopes, and idempotency support for
write calls.

Use this guide for sequencing and behavioral rules. Use the generated endpoint pages under
**Endpoints** for the complete schemas, response types, and playground examples.

## What changed

Historically, these workflows were split across public CloudApi v1 and Angular/internal v1
services. v2 now puts the same operations behind governed bearer-token APIs:

- Opportunity win creates the project through the existing opportunity/project workflow.
- Opportunity billing and site addresses use the same address save workflow, tax refresh, and activity side effects as the app.
- Quote and project item writes call the same quote/project item manager paths the app uses.
- Quote and project location/system create routes are idempotent by name. Item create can assign locations and systems by public GUID, legacy integer ID, or name.
- Payment schedules and invoices use the existing project payment managers.
- Install-status updates remain lifecycle commands, not generic field patches.
- Catalog brand, category, supplier, setup, tax, pricing, and new-product reference reads use the existing CatalogApi v1 managers while omitting legacy row IDs.
- Catalog product create, add-from-library, update, price, barcode, and status write commands use typed v2 DTOs over public product GUIDs.
- Catalog package metadata, package composition, and item accessories use tenant-scoped catalog item GUIDs and atomic replacement commands for migration-safe writes.
- Client, opportunity, and project notes use public parent GUIDs and public note IDs; pinned note commands preserve the existing note manager behavior.
- UI To-Dos for clients, opportunities, and projects use public To-Do and assignee GUIDs and are separate from v2 work items and project task-group APIs.
- GUID-backed file uploads, parent-scoped project/quote location and system uploads, project/service-call invoice uploads, project subscription uploads, proposal uploads, and file link/name/visibility/delete commands use the existing file manager after v2 entity resolution.
- Catalog item images and tenant company logo commands use the existing upload/company managers without accepting internal item or account IDs.
- Purchase-order totals, status summaries, estimates, shipment tracking URLs, create, status, archive, delete, send/email, ADI direct-order, Amazon direct-order, and catalog-cost application actions use the existing purchase-order workflow.
- Service-call address reads/writes and state transitions use the existing ServiceApi manager so tax, visit, payment, inventory, checklist, activity, and webhook side effects stay intact.
- Service-contract aggregates, lifecycle commands, payment schedules, invoices, invoice email, and invoice file upload use the existing ServiceApi v1 paths.
- Registration, anonymous payment-session, and recipient checkout flows stay deferred.

  failure unless you first re-read the purchase order. The same caution applies to service
  contract delete commands.
</Note>

## Before you call

Every non-anonymous v2 call requires:

```http
Authorization: Bearer <access_token>
X-DTools-AccountId: <account-id>
```

The bearer token identifies the platform principal. `X-DTools-AccountId` selects the tenant
context when the token can access more than one account. The API resolves the account into
`PlatformContext`, bridges that context into the legacy `Session`, and then calls the existing
workflow managers.

Writes should include an idempotency key:

```http
Idempotency-Key: win-opportunity-9d6048f4-20260608
```

The key is scoped by tenant, method, path, and key value. Reusing it on the same request lets
the API replay the original successful response. Do not reuse the same key for a different
request body.

## Identifiers

Route IDs are public GUIDs. v2 resolves those GUIDs inside the authenticated tenant before
calling the existing D-Tools workflow managers. This applies to opportunities, projects,
quotes, change orders, project and quote locations/systems, clients, invoices/payments,
payment terms, purchase orders, files, quote items, project items, notes, UI To-Dos,
To-Do assignee users, service calls, service contracts, service-contract payment terms, and
service-contract invoices.

Opportunity billing and site addresses are singleton child resources. They are addressed by
their parent opportunity GUID plus the fixed `billing` or `site` route segment because the
legacy address rows do not have public GUIDs. Address delete by legacy row ID remains
deferred until a public address identity or explicit delete command is available.

Notes use the note `CorrelationId` as the public note ID. UI To-Dos use a public To-Do GUID
and `Users.UniqueId` for assignees. These To-Do routes are not the same as v2 `work-items`
or project task-group/task planning APIs.

Some request-body fields still follow the generated endpoint schema for the existing manager
contract. When a field is a public resource identity in v2, it is represented as a GUID. When
the schema exposes a catalog, status, location, system, phase, or payment status numeric field,
use the values returned by the relevant read/catalog endpoint until a dedicated v2 catalog for
that field is available.

Install statuses are different: they are selected by status/state name rather than public
GUID because the current install-status catalog is state/name based. Partial install means
passing only the project item row GUIDs that should be marked installed. Quantity-level
partial install requires splitting quantities first, then updating the subset of item rows.

## Endpoint details

### Read catalog reference data

```http
GET /api/v2/catalog/brands
GET /api/v2/catalog/brands/new
GET /api/v2/catalog/brands/{brandDtin}
GET /api/v2/catalog/brands/{brandDtin}/suppliers
GET /api/v2/catalog/brands/{brandDtin}/new-products
GET /api/v2/catalog/categories
GET /api/v2/catalog/categories/icons
GET /api/v2/catalog/setup/structure
GET /api/v2/catalog/tax-settings
GET /api/v2/catalog/taxes
GET /api/v2/catalog/pricing-options
GET /api/v2/catalog/suppliers
GET /api/v2/catalog/suppliers/dtins
GET /api/v2/catalog/suppliers/{supplierDtin}
GET /api/v2/catalog/suppliers/{supplierDtin}/exists
GET /api/v2/catalog/suppliers/{supplierDtin}/new-products
POST /api/v2/catalog/products/resolve-dtins
```

These endpoints expose the CatalogApi v1 reference data that Angular previously read
through internal/private API calls. They are read-only in this phase. Brand, supplier,
category, product, parent-category, and new-product identifiers are public DTINs where
the legacy data has a DTIN. Legacy row IDs such as brand ID, supplier ID, category ID,
tax ID, adjustment ID, item-discount ID, length rule ID, and parent row ID are not
returned.

Catalog categories return parent DTINs when the parent category has one. Tax settings
translate configured product and labor tax row IDs into tax names. Group taxes return
child tax names and rates rather than child row IDs. `POST /catalog/products/resolve-dtins`
accepts a list of product DTINs and returns the subset present in the tenant catalog:

```json
{
  "dtins": [
    "88f782df-f23a-4c2f-92cd-771d2b2996c1",
    "8d124f9f-4e77-49ea-99c1-6a60fd2c1228"
  ]
}
```

Use these endpoints as supporting catalogs for quote/project item creation, purchase-order
estimates, and UI setup selectors. Catalog setup writes, generic FieldValue patch paths,
inventory writes, and long-tail tenant lookup-table administration remain separate parity
targets until each workflow has an explicit v2 command contract.

### Create and update catalog products

```http
POST /api/v2/catalog/products
POST /api/v2/catalog/products/from-library
PUT /api/v2/catalog/products/{id}
PUT /api/v2/catalog/products/prices
PUT /api/v2/catalog/products/barcodes
PUT /api/v2/catalog/products/statuses
```

These routes cover catalog product write paths without exposing arbitrary `FieldValue`
patching. Custom product create and product update accept typed product fields and resolve
brand, category, supplier, and manufacturer references internally. Add-from-library accepts
a library product DTIN and optional supplier DTIN, then returns the tenant product public
GUID.

Bulk price, barcode, and status commands use public product GUIDs and delegate to the
existing product service:

```json
[
  {
    "id": "88f782df-f23a-4c2f-92cd-771d2b2996c1",
    "unitPrice": 499,
    "unitCost": 300,
    "msrp": 549
  }
]
```

Category, supplier, setup, and inventory writes remain deferred until those tenant-local
rows have public identity or an approved natural-key command contract.

### Migrate packages and accessories

```http
GET /api/v2/catalog/packages
POST /api/v2/catalog/packages
GET /api/v2/catalog/packages/{id}
PUT /api/v2/catalog/packages/{id}
GET /api/v2/catalog/packages/{id}/items
PUT /api/v2/catalog/packages/{id}/items
GET /api/v2/catalog/items/{id}/accessories
PUT /api/v2/catalog/items/{id}/accessories
POST /api/v2/catalog/items/{id}/accessories/reset-to-category-defaults
```

Create the referenced products, allowances, labor items, and nested packages first. Then
replace the package composition with public catalog item GUIDs:

```json
{
  "items": [
    {
      "itemId": "88f782df-f23a-4c2f-92cd-771d2b2996c1",
      "quantity": 2,
      "laborTime": 1800,
      "laborPrice": 75,
      "isBillable": true
    }
  ]
}
```

Accessories use the same public GUID model and add an inclusion option:

```json
{
  "accessories": [
    {
      "itemId": "1144f44f-2665-405f-a6d8-e91eb715721f",
      "quantity": 1,
      "includeOption": "recommended",
      "isBillable": true
    }
  ]
}
```

Both `PUT` operations atomically replace the complete relationship. Submit an empty
`items` or `accessories` collection to clear it. Duplicate item GUIDs, self-references,
unsupported catalog item types, and nested-package cycles are rejected. The reset command
rebuilds accessories from the item's category defaults. All writes support
`Idempotency-Key`.

### Manage notes

```http
GET /api/v2/clients/{clientId}/notes
POST /api/v2/clients/{clientId}/notes
GET /api/v2/clients/{clientId}/notes/{noteId}
POST /api/v2/clients/{clientId}/notes/{noteId}/pin
POST /api/v2/clients/{clientId}/notes/{noteId}/unpin

GET /api/v2/opportunities/{opportunityId}/notes
POST /api/v2/opportunities/{opportunityId}/notes
GET /api/v2/opportunities/{opportunityId}/notes/{noteId}
POST /api/v2/opportunities/{opportunityId}/notes/{noteId}/pin
POST /api/v2/opportunities/{opportunityId}/notes/{noteId}/unpin

GET /api/v2/projects/{projectId}/notes
POST /api/v2/projects/{projectId}/notes
GET /api/v2/projects/{projectId}/notes/{noteId}
POST /api/v2/projects/{projectId}/notes/{noteId}/pin
POST /api/v2/projects/{projectId}/notes/{noteId}/unpin
```

Note routes are scoped by the owning parent resource. The policies follow that parent:
client notes use `Clients:Read`/`Clients:Write`, opportunity notes use
`Opportunities:Read`/`Opportunities:Write`, and project notes use
`Projects:Read`/`Projects:Write`.

List routes support `page`, `pageSize`, and `pinned`. Create accepts note text and returns
the public note detail. Pin and unpin are explicit commands so clients do not need to submit
legacy note row IDs or internal pinned-note state.

### Manage UI To-Dos

```http
GET /api/v2/clients/{clientId}/todos
POST /api/v2/clients/{clientId}/todos
POST /api/v2/clients/{clientId}/todos/batch
GET /api/v2/opportunities/{opportunityId}/todos
POST /api/v2/opportunities/{opportunityId}/todos
POST /api/v2/opportunities/{opportunityId}/todos/batch
GET /api/v2/projects/{projectId}/todos
POST /api/v2/projects/{projectId}/todos
GET /api/v2/todos/{todoId}
PUT /api/v2/todos/{todoId}
POST /api/v2/todos/{todoId}/complete
```

These routes expose the UI To-Do store. They are governed by `Todos:Read`
and `Todos:Write` in the `crm` family. List routes support `statusId`, `search`, `sort`,
`page`, and `pageSize`. Create and update accept name, description, priority, status, due
date, reminder date, completion fields, and assignee `resourceIds` as user GUIDs.
Client and opportunity batch routes accept a `todos` array of the same create payload,
validate the full batch before writing, and return a synchronous batch result with the
created To-Do IDs.

Example create request:

```json
{
  "name": "Confirm rack location",
  "description": "Call the client before finalizing the equipment list.",
  "priorityId": 2,
  "statusId": 1,
  "dueDate": "2026-06-30T00:00:00Z",
  "resourceIds": [
    "1e1ef4c7-4f8c-4958-8a74-156a8be05ad5"
  ]
}
```

`POST /todos/{todoId}/complete` marks a To-Do complete and can accept an explicit
`completedDate`. This slice does not add delete, attachment upload, notifications, quote
To-Do, location To-Do, system To-Do, or first-class checklist write endpoints.

### Upload a file

```http
POST /api/v2/files
Content-Type: multipart/form-data
```

Uploads a file through the existing file upload manager after resolving the target entity's
public GUID inside the authenticated tenant. The multipart form fields are:

| Field | Required | Description |
| --- | --- | --- |
| `entityType` | Yes | Supported values: `client`, `opportunity`, `project`, `quote`, `product`, `subscription`, `purchase-order`, `service-call`, `service-contract`. |
| `entityId` | Yes | Public GUID for the target entity. |
| `type` | No | Optional legacy file type/category string. |
| `file` | Yes | Binary file part. |

Example:

```bash
curl -X POST "https://dtoolsdevapi.azure-api.net/api/v2/files" \
  -H "Authorization: Bearer $TOKEN" \
  -H "X-DTools-AccountId: $ACCOUNT_ID" \
  -H "Idempotency-Key: file-upload-20260608-001" \
  -F "entityType=project" \
  -F "entityId=7a88f502-0d4d-4bd4-b195-f00c9c163dd3" \
  -F "type=Document" \
  -F "file=@./proposal.pdf"
```

The response is the v2 public file detail shape. It uses the file public GUID, not the legacy
integer file ID returned by the internal upload manager. Use this endpoint for root-level
entity attachments.

### Upload catalog images and company logos

```http
POST /api/v2/catalog/items/{itemId}/images
POST /api/v2/company/logo
DELETE /api/v2/company/logo
Content-Type: multipart/form-data
```

`POST /catalog/items/{itemId}/images` uploads an image to a catalog item after resolving
the public item GUID to the tenant-scoped catalog row. The image part must be named
`image`; optional form fields are `typeId` and `order`, matching the legacy image upload
contract. The response returns public image URL metadata and suppresses the legacy
item-image row ID:

```bash
curl -X POST "https://dtoolsdevapi.azure-api.net/api/v2/catalog/items/88f782df-f23a-4c2f-92cd-771d2b2996c1/images" \
  -H "Authorization: Bearer $TOKEN" \
  -H "X-DTools-AccountId: $ACCOUNT_ID" \
  -H "Idempotency-Key: catalog-item-image-20260609-001" \
  -F "typeId=1" \
  -F "order=0" \
  -F "image=@./front.jpg"
```

`POST /company/logo` and `DELETE /company/logo` are singleton tenant commands. They do
not accept an account ID in the route or body; v2 derives the tenant from `PlatformContext`
and delegates to the existing company logo manager. The upload part must be named `logo`,
and the upload response returns the logo URL.

### Upload location and system files

### Upload invoice, subscription, and proposal files

```http
POST /api/v2/projects/{projectId}/invoices/{invoiceId}/files
POST /api/v2/service-calls/{serviceCallId}/invoices/{invoiceId}/files
POST /api/v2/service-contracts/{serviceContractId}/invoices/{invoiceId}/files
POST /api/v2/projects/{projectId}/subscriptions/{subscriptionId}/files
POST /api/v2/quotes/{quoteId}/proposal/files
POST /api/v2/projects/{projectId}/proposal/files
POST /api/v2/projects/{projectId}/change-orders/{changeOrderId}/proposal/files
Content-Type: multipart/form-data
```

These routes cover additional Angular/internal v1 upload workflows without exposing legacy
payment, project-subscription, or proposal row IDs. v2 resolves the public parent GUID first,
then resolves the child resource inside that parent:

| Route family | Parent resolution | Child resolution | Response |
| --- | --- | --- | --- |
| Project invoice files | `projectId` public project GUID | `invoiceId` public payment GUID linked to the project payment terms | Public v2 file detail |
| Service-call invoice files | `serviceCallId` public service-call GUID | `invoiceId` public payment GUID linked to the service call | Public v2 file detail |
| Service-contract invoice files | `serviceContractId` public service-contract GUID | `invoiceId` public payment GUID linked to the service contract | Public v2 file detail |
| Project subscription files | `projectId` public project GUID | `subscriptionId` public project-subscription GUID under the project | Public v2 file detail |
| Proposal files | Quote, project, or project/change-order public GUIDs | Internal proposal row resolved from the parent | Legacy proposal file detail |

Invoice upload routes preserve the v1 `ManagePayments` permission check inside the legacy
file-upload facade. Generic catalog subscription uploads through `POST /api/v2/files` with
`entityType=subscription` preserve the v1 `EditCatalogItem` check. Proposal uploads preserve
the v1 `Internal = true` file behavior, so their response is the legacy proposal file detail
instead of the public v2 file detail; internal proposal file rows are not returned by public
file reads.

Multipart fields:

| Field | Required | Description |
| --- | --- | --- |
| `type` | No | Optional legacy file type/category string for invoice and project-subscription uploads. Proposal uploads ignore this value. |
| `file` | Yes | Binary file part. |

## Manager behavior

These endpoints are v2 controllers over the existing workflow managers. They do not implement
parallel business logic. Existing manager permissions, locked-change-order checks, inventory
guards, opportunity address tax refresh and activity creation, project payment term side
effects, invoice status rules, activities, notifications, file-upload rules, purchase-order
side effects, service-call address/state side effects, service-contract side effects, and
webhooks still run downstream.

Direct approved-project item add is intentionally not exposed. Add items to approved projects
through a change order so the same locked-state and inventory behavior used by the D-Tools app
is preserved.

Supplier ordering discovery, purchase-order item write/delete workflows, catalog setup writes,
inventory writes, drawing/task/service-plan uploads, and long-tail tenant settings remain
planned separately because those v1 paths still expose internal child IDs, generic patch
contracts, or token-specific/anonymous semantics.

## Error handling

All errors use the shared `V2ErrorEnvelope` contract. Common codes include:

| Status | Typical cause |
| --- | --- |
| `400 invalid_argument` | Bad GUID format, missing required body, missing item/payment term IDs, or invalid status input. |
| `401 unauthorized` | Missing or invalid bearer token. |
| `403 permission_missing` | The principal lacks the required policy, such as `Projects:Write`. |
| `404 not_found` | The route GUID does not resolve inside the selected tenant. |
| `409 conflict` | Manager validation failed because the workflow state does not allow the requested change. |
| `429 budget_exceeded` | Platform budget/rate policy blocked the request. |
| `503 feature_unavailable` | Release gate or platform configuration makes the endpoint unavailable for this tenant/principal. |

Include the `correlationId` from the error response when contacting support.

## Registration

Registration, anonymous payment-session, and recipient checkout workflows are deferred from
this parity phase. They remain outside the normal bearer-token v2 surface until their auth and
recipient-token model is split from the standard platform API.
